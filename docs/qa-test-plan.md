# QA automation test plan

## Scope and audit findings

Ledgerline supports anonymous document intake, deterministic PDF/TXT/JSON extraction, confidence scoring, review corrections, approval, duplicate detection, audit history, dead-letter retry, CSV export, and optional credential-backed Google integrations. The UI and API contain no login screen, user store, authentication middleware, roles, or protected routes. Authentication, authorization, and role-permission scenarios are therefore documented gaps rather than fabricated tests.

The existing baseline is 15 Pytest tests and 3 Vitest tests. GitHub Actions previously ran backend lint/tests/coverage and frontend lint/tests/build. Docker Compose builds Nginx/React, FastAPI, and PostgreSQL; health checks now make readiness explicit. Sanitized PDFs in `sample-data/synthetic-pdfs/` are deterministic and can be regenerated by `scripts/generate_synthetic_pdfs.py`.

Determinism risks found during audit:

- PostgreSQL data persists in the named volume, so repeated static fixtures become duplicates unless the test stack starts with `docker compose down --volumes`.
- Record timestamps and UUIDs are generated at runtime; assertions intentionally avoid exact values.
- The UI sorts newest-first, so tests locate business identifiers instead of row positions.
- Google Sheets and Gmail require optional external credentials; E2E tests exclude them and verify only the local configuration error where relevant.
- The product has no database reset API. The suite uses one Chromium worker and an isolated Compose volume per run.
- Docker's first image build can be slow; CI uses Compose health checks and a bounded job timeout.

## Coverage matrix

| Area | Layer | Evidence |
| --- | --- | --- |
| Compose startup/readiness | UI/API | Dashboard load; health endpoint; service health checks |
| Login/authentication | Gap | Not implemented by the application |
| Authorization/permissions | Gap | All routes are anonymous; no roles or ownership model |
| Valid PDF upload | UI/API | Approved invoice and receipt extraction |
| Unsupported file | UI/API negative | 415 and user-visible message |
| Malformed/incomplete input | API negative + existing Pytest | 422 approval validation; dead-letter parser coverage |
| Extracted fields/confidence | UI/API | Vendor, reference, status, aggregate and field scores |
| Reviewer workflow | UI/API | Correction, approval, actor, audit sequence |
| Duplicate behavior | UI + existing Pytest | Duplicate status/link; approval conflict covered in backend tests |
| Search/filter | UI | Approved filter plus vendor search |
| CSV export | UI + existing Pytest | Browser download and backend content assertions |
| Error states | UI/API | 415, 422, and 404 responses |
| Accessibility | UI | axe WCAG 2/2.1 A and AA automated scan |
| Failure evidence | Framework | Screenshot, trace, and video retained on failure |
| Reporting | Framework/CI | HTML report and 14-day Actions artifact |

## Test inventory

- 6 direct API tests

- 1 axe accessibility test

## Local execution

Requirements: Docker Desktop, Node.js 22+, and npm.

```powershell
npm ci
npx playwright install chromium
docker compose down --volumes --remove-orphans
docker compose up --build --wait --wait-timeout 180
npm run test:e2e
npm run test:e2e:report
```

The destructive-looking `down --volumes` command removes only this Compose project's disposable PostgreSQL test volume. It is required before each run because fixture uploads are intentionally idempotent and would otherwise appear as duplicates.

For debugging, use `npm run test:e2e:ui`. Override targets with `PLAYWRIGHT_BASE_URL`, `PLAYWRIGHT_API_URL`, and `PLAYWRIGHT_API_ORIGIN`. The supported default is the local Compose stack.

## Exit criteria

- Legacy Pytest, Vitest, lint, and build checks remain green.
- All Playwright projects pass against a fresh Compose volume.
- The HTML report is generated.
- No retries are needed locally; any CI retry is visible in the report.
- Remaining product gaps and any flaky behavior are reported rather than hidden.
